## 2. 論文の問題設定（定義）
`main.tex:382-425` に基づく。

- `diff`: CIに投稿されるコード変更。
- `catching test`: 生成時点で対象diffに対して失敗するよう設計されたテスト。
- `weak catch`: 親リビジョンでは通り、対象diffで失敗するテスト。
- `strong catch`: weak catch のうち、実際にバグを示す真陽性失敗。
- 課題: weak catch を大量に得つつ、false positiveによる開発速度低下を抑えること。

## 5. 論文の主要結論（数値付き）
`main.tex:433-457`, `main.tex:725-746`, `main.tex:767-794`, `main.tex:1444-1453` に基づく。

- diff-aware は hardening系より weak catch 効率が高い。
  - 論文の要約主張: hardening比で約4倍、coincidental比で約20倍。
- intent-aware は dodgy diff より、少ないテストで高い weak catch率を示す（6.4% vs 2.5%）。
- weak catchを得られたdiff率は intent-aware が 7.9% で最も高い。
- 運用照会 41 件中、8 件が strong catch と確認。
- 8件中4件は重大障害につながる不具合の回避と報告。
- false positive対応は「エンジニアへ期待変更かを短時間確認する」運用で摩擦を低減。

## 6. 判定器（false positive削減）の構成
`main.tex:890-917`, `main.tex:943-986`, `main.tex:995-1104` に基づく。

### 6.1 LLM-as-Judge
- モデル例: Llama3.3-70B, Gemini 3 Pro, Claude Sonnet 4。
- 入力:
  - 生成テストコード
  - 実行結果（失敗メッセージ・トレース）
  - 親子diff
  - 推定diff intent
- 出力:
  - TP確率スコア（-1〜1）
  - アンサンブル分類スコア（High/Medium/Lowを数値化）
  - 説明文
- 論文記載: 閾値運用により人手レビューを約70%削減。

### 6.2 ルールベース（RubFake）
- false positiveパターンとtrue positiveパターンを規則化。
- 例（FP）: broken test runner, reflection, bad mock, infra失敗など。
- 例（TP）: unexpected key change, changed bool, monotonic change, RBAC関連など。
